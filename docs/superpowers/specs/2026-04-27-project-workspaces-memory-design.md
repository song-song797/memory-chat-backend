# 项目工作空间与记忆分层设计

## 目标

为聊天应用增加“项目工作空间”能力，让用户可以把具体项目的聊天、项目记忆和项目设置组织在一起，同时保留普通聊天模式和跨项目生效的全局偏好。

业务目标是降低长期记忆管理成本。用户不应该手动维护大量细碎上下文；系统需要根据聊天所在上下文，把稳定的用户偏好、项目事实和临时会话内容分层处理。这样用户在项目内工作时，助手能继承全局偏好，又不会把其他项目的细节带进当前回答。

## 当前状态

后端已经有：

- `users`：用户。
- `conversations`：聊天会话。
- `messages`：消息。
- `memories`：长期记忆。
- `/api/conversations`：会话 CRUD。
- `/api/memories`：长期记忆 CRUD。
- `/api/chat`：聊天接口，当前会注入启用中的长期记忆。

前端已经有：

- 侧边栏聊天列表。
- 设置抽屉。
- 简单的长期记忆管理区域。

现在的问题是：`memories.kind = "project"` 只是分类，不是真正的项目边界。聊天、记忆和设置还不能围绕项目形成独立工作空间。

## 设计选择

采用“项目作为工作空间，全局记忆仍保留”的方案。

核心规则：

- 新增 `projects` 表，每个用户可以有多个项目。
- `conversations.project_id` 可为空。
- `memories.project_id` 可为空。
- `project_id = null` 的聊天是普通聊天。
- `project_id = null` 的记忆是全局记忆。
- 有 `project_id` 的聊天和记忆属于对应项目。
- 聊天时注入全局 active 记忆，再注入当前项目 active 记忆，项目记忆优先于全局记忆。
- 普通聊天存在且继续可用，但普通聊天里的自动记忆提取必须非常保守。

不拆分 `global_memories` 和 `project_memories`。第一版继续使用一张 `memories` 表，通过 `scope`、`project_id`、`kind`、`status` 和 `superseded_by_id` 表达边界、分层和覆盖关系。全局记忆和项目记忆的生命周期相似，都需要创建、归档、覆盖、注入和以后可能的 embedding 检索；拆表会让 CRUD、候选系统和前端管理重复。

## 数据模型

### `projects`

新增 `Project` 模型：

- `id`：字符串主键。
- `user_id`：所属用户。
- `name`：项目名称。
- `description`：可选项目说明。
- `default_model`：可选项目默认模型。
- `default_reasoning_level`：可选项目默认推理强度。
- `is_default`：是否为系统创建的默认项目。
- `archived_at`：可选归档时间。
- `created_at`：创建时间。
- `updated_at`：更新时间。

每个用户可以有一个默认项目，UI 上可称为“个人空间”。默认项目主要承接旧项目类数据，不承担全局偏好职责。

第一版项目设置只包含名称、描述、默认模型和默认推理强度。项目规则、业务约束和回答要求放在项目记忆的 `rule` 或 `decision` 类型里，不放进设置字段，避免同一类信息有两套来源。

### `conversations`

新增字段：

- `project_id`：可空外键，指向 `projects.id`。

语义：

- `project_id = null`：普通聊天。
- `project_id != null`：项目聊天，挂在对应项目下。

所有会话读取和修改必须校验项目归属和用户归属一致。

### `memories`

新增字段：

- `project_id`：可空外键，指向 `projects.id`。
- `scope`：`global` 或 `project`。
- `status`：`active` 或 `archived`。
- `importance`：整数权重，默认值用于排序。
- `superseded_by_id`：可空外键，指向覆盖它的新记忆。
- `archived_at`：可选归档时间。

扩展 `kind`：

- `preference`：用户偏好。
- `tool`：长期工具偏好。
- `profile`：用户稳定身份或背景。
- `project_overview`：项目目标、产品定位、业务背景。
- `tech_stack`：语言、框架、数据库、部署方式。
- `architecture`：模块边界、数据流、关键服务。
- `decision`：已确认的产品或技术决策。
- `rule`：项目内必须遵守的规则。
- `task_context`：近期任务背景，可被后续覆盖归档。
- `fact`：普通事实。

一致性规则：

- `scope = global` 时，`project_id` 必须为空。
- `scope = project` 时，`project_id` 必须存在。
- 只有 `status = active` 的记忆会注入聊天上下文。
- 自动清理不直接永久删除记忆，而是归档旧记忆并设置 `superseded_by_id`。

## 记忆分层

### 全局层

全局记忆跨普通聊天和所有项目生效，数量应保持克制。适合保存：

- 语言偏好。
- 回答风格。
- 常用工具偏好。
- 用户长期工作习惯。
- 稳定身份或背景。

不适合保存：

- 某个项目的技术栈。
- 某个项目的当前任务。
- 短期调试状态。
- 没有明确长期价值的临时事实。

### 项目层

项目记忆只在当前项目内生效。适合保存：

- 项目目标和背景。
- 技术栈。
- 架构和数据流。
- 业务规则。
- 已确认决策。
- 项目内偏好。
- 近期但仍有用的任务上下文。

项目记忆可以覆盖全局记忆。例如全局偏好是“默认用 Python”，但当前项目记忆说明“本项目使用 TypeScript”，项目内聊天应优先遵守 TypeScript。

### 会话层

会话层来自当前聊天最近消息。它表示用户正在进行的当前任务，优先级最高，但不天然成为长期记忆。

普通聊天的会话层不属于任何项目。项目聊天的会话层只属于对应项目。

## 聊天上下文注入

项目聊天时：

1. 读取全局 active 记忆。
2. 读取当前项目 active 记忆。
3. 按记忆层级和类型组织上下文。
4. 在 system/context 文案中明确项目记忆优先于全局记忆。
5. 读取当前项目下当前会话的最近消息。
6. 调用模型。

普通聊天时：

1. 读取全局 active 记忆。
2. 不读取任何项目记忆。
3. 读取当前普通会话最近消息。
4. 调用模型。

第一版不做复杂语义冲突检测。先通过注入顺序和 system 文案表达优先级。后续 AI 候选系统可以再加入冲突识别和自动归档。

## 记忆写入与自动归档

显式“请记住”请求：

- 如果当前在项目聊天中，默认创建项目记忆。
- 如果用户明确说这是跨项目偏好，创建全局记忆。
- 如果当前是普通聊天，只在内容明显是稳定跨项目偏好时创建全局记忆。
- 如果普通聊天里出现项目事实，不自动写入全局记忆。

隐式提取或候选保存：

- 项目聊天中提取的项目事实进入当前项目记忆或候选。
- 普通聊天中的项目事实不直接进入全局记忆，可以作为未归属候选留给后续版本处理。
- 全局记忆只接受稳定、跨项目、长期有效的信息。

覆盖和清理：

- 新记忆和旧记忆冲突时，优先保留更近、更具体、更明确的记忆。
- 被覆盖记忆设置为 `status = archived`。
- 被覆盖记忆设置 `superseded_by_id` 指向新记忆。
- 第一版不做自动硬删除。

## API 设计

新增 `/api/projects`：

- `GET /api/projects`：列出当前用户项目，默认不返回已归档项目，支持参数包含归档项目。
- `POST /api/projects`：新建项目。
- `PUT /api/projects/{project_id}`：更新名称、描述、默认模型、默认推理强度、归档状态。

第一版不提供硬删除项目接口。归档项目可以隐藏在主界面之外，但数据保留。

调整 `/api/conversations`：

- `GET /api/conversations`：继续返回所有聊天，包含 `project_id`。
- `GET /api/conversations?project_id=...`：可按项目筛选，供后续局部 UI 使用。
- `POST /api/conversations`：可带 `project_id`。不带则创建普通聊天。
- `PUT /api/conversations/{conversation_id}`：可用于重命名、置顶，第一版暂不要求支持跨项目移动。

调整 `/api/chat`：

- 请求体增加 `project_id`。
- 如果 `conversation_id` 存在，以会话已有 `project_id` 为准。
- 如果新建聊天且传入 `project_id`，创建项目聊天。
- 如果新建聊天且未传 `project_id`，创建普通聊天。
- 如果 `conversation_id` 和 `project_id` 冲突，返回 400，避免串项目上下文。

调整 `/api/memories`：

- `GET /api/memories?scope=global`：返回全局记忆。
- `GET /api/memories?scope=project&project_id=...`：返回指定项目记忆。
- `POST /api/memories`：支持创建全局记忆或项目记忆。
- `PUT /api/memories/{memory_id}`：支持更新内容、类型、启用状态或归档状态。
- `DELETE /api/memories/{memory_id}`：第一版可保留现有删除能力，但 UI 主路径优先使用归档/停用。

## 前端设计

侧边栏继续作为所有聊天的总入口，不按当前项目隐藏其他聊天。

展示方式：

- 普通聊天显示在“普通聊天”区域。
- 项目显示为可展开分组。
- 项目下展示该项目聊天。
- 置顶聊天可以继续在顶部区域显示，但需要显示所属项目或普通聊天来源。
- 点击项目聊天后，聊天自动进入该项目上下文。
- 点击普通聊天后，聊天不带项目上下文。

项目入口：

- 支持新建项目。
- 支持重命名项目。
- 支持设置项目默认模型和默认推理强度。
- 支持归档项目。
- 第一版不做硬删除项目。

记忆管理：

- 设置抽屉中的长期记忆拆成“全局记忆”和“当前项目记忆”。
- 普通聊天打开设置时，只展示全局记忆，并提示当前聊天没有项目记忆。
- 项目聊天打开设置时，展示全局记忆和当前项目记忆。
- 新增记忆默认根据当前聊天上下文决定范围：
  - 项目聊天默认新增项目记忆。
  - 普通聊天默认新增全局记忆，但文案应提示只适合稳定偏好。
- 已归档或被覆盖的记忆默认隐藏，可以提供一个小入口查看。

## 旧数据迁移

迁移策略：

- 为每个已有用户创建一个默认项目，UI 名称为“个人空间”。
- 旧聊天归到“个人空间”。
- 旧记忆按 `kind` 迁移：
  - `preference`、`tool`：保持全局记忆。
  - `project`：归到“个人空间”项目记忆，类型可映射为 `project_overview` 或 `fact`。
  - `fact`：保持全局记忆，后续可被系统覆盖或归档。
- 迁移后，新普通聊天仍允许 `project_id = null`。

这样旧用户能继续看到历史聊天，同时全局偏好不会被错误塞进默认项目。

## 错误处理

- 项目不存在或不属于当前用户时返回 404。
- 会话和项目不属于同一用户时返回 404。
- `conversation_id` 与请求中的 `project_id` 冲突时返回 400。
- 项目聊天读取项目记忆失败时，聊天请求可以降级为只使用全局记忆和会话上下文，并记录错误。
- 全局记忆读取失败时，聊天请求继续，但不注入全局记忆，并记录错误。
- 自动归档失败不能阻止聊天回复。

## 测试

后端测试需要覆盖：

- 项目 CRUD 校验用户归属。
- 新建普通聊天时 `project_id` 为空。
- 新建项目聊天时 `project_id` 正确保存。
- 会话列表返回所有聊天，并包含项目归属。
- 项目筛选只返回该项目聊天。
- 聊天请求中 `conversation_id` 和 `project_id` 冲突会失败。
- 普通聊天只注入全局记忆。
- 项目聊天注入全局记忆和当前项目记忆。
- 项目聊天不注入其他项目记忆。
- 项目记忆在上下文中优先于全局记忆。
- 旧数据迁移把旧聊天归到个人空间。
- 旧偏好类记忆保持全局。

前端第一版没有稳定测试体系，至少需要手动验证：

- 创建项目。
- 在项目下创建聊天。
- 普通聊天继续可用。
- 侧边栏能同时看到普通聊天和项目聊天。
- 项目聊天中的记忆管理显示当前项目记忆。
- 普通聊天不会误导用户把项目事实保存为全局偏好。

## 非目标

第一版不做：

- 项目硬删除。
- 聊天在项目之间移动。
- 复杂语义冲突检测。
- embedding 或 pgvector。
- 自动把普通聊天里的项目事实分配到项目。
- 项目级权限协作。

## 后续增强

后续可以加入：

- 聊天移动到项目。
- 未归属记忆候选。
- AI 自动识别冲突并归档旧记忆。
- 项目级设置。
- 项目归档后的恢复界面。
- embedding 检索，让全局和项目记忆按相关性注入。
